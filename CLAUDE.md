## 1. 背景と目的

### 1.1 現行サイトの概要

千葉大学 大学院融合理工学府 先進理化学専攻 物質科学コースに所属する「量子ナノデバイス研究グループ（青木研究室）」の公式ホームページである。2018年4月に作成された静的HTMLサイトで、日本語版・英語版の2言語構成を持つ。

現行サイトのページ構成は以下の通りである。

- **トップページ**（index.html / index-e.html）：研究グループの概要紹介、最新ニュース・論文情報の時系列掲載
- **研究内容**（research.html）：ファンデルワールスヘテロ接合、量子ポイントコンタクト・量子ドット、TMDデバイス、走査プローブ顕微法などの研究テーマ詳細
- **メンバー**（members.html / members-e.html）：教職員・学生一覧、卒業生進路先
- **研究設備**（facility.html）：クリーンルーム・試料作製装置・評価設備の紹介
- **業績**（publication関連ページ）：論文リスト・学会発表
- **アクセス**（アクセス情報ページ）

### 1.2 現行サイトの課題

現行サイトには以下の構造的・技術的課題が存在する。

- **レスポンシブ非対応：** モバイル端末での閲覧性が著しく低い。大学ホームページの訪問者の50%以上がモバイルからアクセスする現状に対応できていない。
- **更新の属人化：** HTMLファイルを直接編集する必要があり、Web制作スキルを持たない研究室メンバーにとって更新のハードルが高い。2021年以降の情報更新が滞っている。
- **SEO未対応：** メタタグ、構造化データ、OGPタグがなく、検索エンジンからの発見性が低い。
- **デザインの陳腐化：** 2018年当時のデザインのまま、視覚的な訴求力やブランディングが不足している。
- **アクセシビリティ未対応：** alt属性の欠如、コントラスト比の未検証、キーボードナビゲーション非対応。
- **パフォーマンス：** 画像の未最適化、不要なインラインスタイル、HTTP/2未対応の可能性。

### 1.3 再開発の目的

現行コンテンツの内容を忠実に維持しつつ、以下の目的を達成する。

- **保守性の向上：** 研究室メンバーが専門知識なしにコンテンツを更新できる仕組みの構築
- **ユーザー体験の改善：** レスポンシブ対応、モダンなUI、高速な表示速度の実現
- **国際的な研究室ブランディング：** 海外研究者・留学希望者に向けた情報発信力の強化
- **長期運用可能なアーキテクチャ：** メンバー交代があっても持続的に運用可能な技術基盤の選定

## 2. 要件

### 2.1 情報構造（サイトマップ）

再開発後のサイトは以下の階層構造で構成する。各ページは日本語版・英語版を用意し、言語切替UIをヘッダーに配置する。

```
/（トップページ）
├── /research（研究内容）
│   ├── ファンデルワールスヘテロ接合
│   ├── 量子デバイス（量子ポイントコンタクト・量子ドット）
│   ├── TMDデバイス（MoTe2 ドーピング・インバータ）
│   └── 走査プローブ顕微法・ナノ加工
├── /members（メンバー）
│   ├── 教職員
│   ├── 学生
│   └── 卒業生・進路先
├── /publications（業績）
│   ├── 論文一覧（年度別フィルタ機能付き）
│   └── 学会発表
├── /facilities（研究設備）
│   ├── クリーンルーム・試料作製装置
│   └── 評価設備
├── /news（ニュース・お知らせ）
├── /access（アクセス・連絡先）
└── /en/（英語版ルート、同一構造のミラー）
```

### 2.2 主要機能要件

**F-01 多言語対応（日本語・英語）**  
i18nルーティングにより、`/ja/` と `/en/` のパスベースで言語切替を行う。ヘッダーのトグルボタンで即時切替可能とする。コンテンツは言語別のMarkdownファイルまたはJSONで管理する。

**F-02 ニュース・お知らせ機能**  
トップページに最新ニュース5件を表示し、「ニュース一覧」ページへのリンクを設ける。各ニュースにはカテゴリタグ（論文発表、受賞、イベント、メンバー異動など）を付与し、フィルタリング可能とする。ニュースの追加はMarkdownファイルの作成のみで完了する設計とする。

**F-03 論文・業績データベース**  
BibTeX形式またはYAML/JSONでの論文データ管理を行い、年度別・著者別・キーワード別のフィルタ・検索機能を提供する。DOIリンクの自動生成と、Google Scholar / ORCIDへの外部リンク設置を行う。

**F-04 メンバー管理**  
YAML/JSONファイルによるメンバーデータ管理。名前、役職、研究テーマ、メールアドレス（_at_方式によるスパム対策）、写真を含む。卒業生の進路先一覧も同一データソースから生成する。

**F-05 研究設備ギャラリー**  
設備写真のギャラリー表示。Lightbox形式の拡大表示対応。各設備に名称・用途の説明テキストを添付する。

**F-06 アクセス情報**  
キャンパスマップの埋め込み（Google Maps API またはOpenStreetMap）、最寄駅からの経路案内テキスト、連絡先情報の表示。

**F-07 レスポンシブWebデザイン**  
デスクトップ（1200px以上）、タブレット（768px〜1199px）、モバイル（767px以下）の3ブレークポイントで最適化されたレイアウトを提供する。

### 2.3 ユーザー視点での期待動作

本サイトの主要ユーザーペルソナと期待動作を以下に整理する。

**ペルソナA：大学院進学希望の学部生（国内）**  
研究内容の概要を短時間で理解したい。メンバー構成と卒業生の進路先を確認し、研究室選びの判断材料にしたい。トップページ → 研究内容 → メンバー → アクセスの導線を3クリック以内で完結させる。

**ペルソナB：海外の共同研究者・留学希望者（英語話者）**  
英語ページで研究概要と最新論文を確認したい。コンタクト先へ迷わずたどり着きたい。英語トップ → Research → Publications → Accessの導線を提供する。

**ペルソナC：研究室の教員・学生（管理者）**  
論文掲載やメンバー異動の際に、コードを書かずにサイトを更新したい。Markdownファイルの追加やYAMLの編集だけで反映される仕組みを求める。

**ペルソナD：学会・企業関係者**  
研究設備の有無や研究テーマの詳細を確認し、共同研究の可能性を探りたい。設備一覧と論文リストへ素早くアクセスしたい。

## 3. UX/UI設計

### 3.1 デザインコンセプト

「**Clean Academic — 信頼性と先端性の融合**」をコンセプトとする。量子ナノデバイス研究の先端性を視覚的に表現しつつ、大学研究室としての信頼感と落ち着きを両立するデザインを目指す。

- **カラーパレット：** 千葉大学のブランドカラー（緑系 #006633）をプライマリカラーとし、ダークネイビー（#1a1a2e）をアクセント、白（#ffffff）とライトグレー（#f5f7fa）をベースとする。研究テーマのナノスケール感を表現するグラデーション要素を控えめに使用する。
- **タイポグラフィ：** 日本語は「Noto Sans JP」、英語は「Inter」をWebフォントとして使用。見出しは太字（700）、本文はレギュラー（400）で統一する。
- **ビジュアル要素：** ヒーローセクションにSEM像や量子デバイスの顕微鏡写真を大きく配置し、研究の魅力を直感的に伝える。

### 3.2 レイアウト案（画面構成）

**共通ヘッダー（全ページ固定）：**  
左にロゴ（研究室名＋千葉大学ロゴ）、中央にグローバルナビゲーション（トップ、研究内容、メンバー、業績、設備、ニュース、アクセス）、右に言語切替トグル（JP / EN）を配置する。モバイル時はハンバーガーメニューに折りたたむ。

**トップページ：**  
ヒーローセクション（研究イメージ写真＋キャッチコピー＋「研究内容を見る」CTA）→ 研究概要セクション（3〜4つの研究テーマをカード形式で表示）→ 最新ニュースセクション（5件のタイムライン表示）→ アクセス・コンタクト情報のサマリ → フッター。

**研究内容ページ：**  
左サイドにサブナビゲーション（研究テーマ一覧）、メインエリアに各テーマの詳細説明（テキスト＋図版）を配置する。モバイル時はサブナビがアコーディオン形式に変化する。

**メンバーページ：**  
教職員セクション（写真＋名前＋役職＋研究テーマ＋メールのカード形式）→ 学生セクション（リスト形式）→ 卒業生・進路先セクション（企業名リスト）の3段構成。

**業績ページ：**  
年度セレクタとカテゴリフィルタを上部に配置。論文リストはカード形式で、タイトル、著者、ジャーナル名、年、DOIリンクを表示する。

**研究設備ページ：**  
グリッドレイアウトで設備写真を一覧表示。クリックでLightbox拡大。カテゴリタブ（作製装置 / 評価設備）で切り替え可能。

**共通フッター：**  
千葉大学ロゴ、研究室連絡先、所在地、関連リンク（大学本部・専攻ページ等）、Copyright表記を配置する。

### 3.3 画面遷移図（テキスト記述）

```
[トップページ]
  ├──→ [研究内容] ──→ [各研究テーマ詳細（ページ内アンカー or サブページ）]

※ 全ページからヘッダーナビにより任意のページへ直接遷移可能
```

### 3.4 レスポンシブ対応

- **デスクトップ（≥1200px）：** 2〜3カラムレイアウト。サイドナビゲーション表示。ヒーロー画像フル幅。
- **タブレット（768px〜1199px）：** 2カラムからシングルカラムへの段階的折りたたみ。ナビゲーションは横並び維持。
- **モバイル（<768px）：** シングルカラム。ハンバーガーメニュー。タッチフレンドリーなタップターゲット（最小44×44px）。画像は遅延読み込み（lazy loading）適用。

### 3.5 アクセシビリティ配慮

- WCAG 2.1 Level AA準拠を目標とする。
- すべての画像にalt属性を付与する。研究画像には「SEM image of quantum point contact device」のような具体的な説明を記述する。
- カラーコントラスト比4.5:1以上を確保する。
- キーボードのみでの全機能操作を保証する（Tab/Enter/Escapeキー対応）。
- スクリーンリーダー対応のためのARIAランドマーク（nav, main, footer）とaria-label属性を適切に配置する。
- フォーカスインジケータの視認性を確保する。

## 4. 技術的アプローチ

### 4.1 技術スタック

| レイヤー | 技術選定 | 選定理由 |
|---|---|---|
| 静的サイトジェネレータ | **Hugo** | ビルド速度が極めて高速（数百ページを数秒で生成）。Go製のシングルバイナリで環境構築が容易。多言語サイト（i18n）のネイティブサポート。大学研究室の規模に最適。 |
| テンプレートエンジン | Go Templates（Hugo内蔵） | Hugo標準。追加依存なし。 |
| CSS フレームワーク | **Tailwind CSS** | ユーティリティファーストで一貫したデザインを維持しやすい。PurgeCSSによる未使用CSS自動削除でファイルサイズ最小化。 |
| コンテンツ管理 | **Markdown + YAML Front Matter** | 研究室メンバーがテキストエディタで編集可能。Gitでの変更管理と相性が良い。 |
| データファイル | **YAML / JSON** | メンバー情報・論文リスト・設備データなどの構造化データを管理。Hugo Data Templatesで自動レンダリング。 |
| バージョン管理 | **Git + GitHub** | コンテンツの変更履歴管理。プルリクエストによるレビューフロー。 |
| ホスティング | **GitHub Pages** または **Cloudflare Pages** | 無料で高可用性。CDN配信による高速表示。SSL/TLS自動対応。大学ドメインのCNAME設定で `adv.chiba-u.jp` サブパスへの割り当てが可能。 |
| CI/CD | **GitHub Actions** | mainブランチへのpush時に自動ビルド・デプロイ。プレビュー環境のPR毎自動生成。 |
| 画像最適化 | **Hugo Pipes + WebP変換** | 画像のリサイズ・WebP変換をビルド時に自動実行。 |
| 検索機能 | **Pagefind**（静的検索） | ビルド時にインデックスを生成するクライアントサイド検索。外部サービス不要。日本語対応。 |

### 4.2 静的サイトジェネレータ（Hugo）の選定理由

WordPressなどの動的CMSではなく、静的サイトジェネレータを採用する理由は以下の通りである。

- **セキュリティ：** 静的ファイル配信のため、SQLインジェクションやXSSなどの攻撃ベクトルが存在しない。サーバー管理の負担がゼロ。
- **コスト：** GitHub PagesやCloudflare Pagesを利用すればホスティング費用が無料。大学研究室の予算制約に適合する。
- **パフォーマンス：** CDN経由の静的配信により、世界中どこからでも高速にアクセス可能。Lighthouseスコア90以上を目標とする。
- **保守性：** Node.jsやPHPのランタイム更新、DBパッチ適用といった運用コストが発生しない。
- **Hugoを選定した追加理由：** Jekyll（Ruby）やNext.js（Node.js）と比較して、Hugoはシングルバイナリで導入が最も簡単であり、多言語サイトのネイティブ機能が充実している。研究室のメンバーが交代しても環境構築でつまずくリスクが最小。

### 4.3 SEO対策

- 各ページにtitle、meta description、OGPタグ（og:title, og:description, og:image）を自動生成する。
- JSON-LD形式の構造化データ（Organization, ResearchProject, Person）を埋め込み、Google検索でのリッチリザルト表示を狙う。
- 論文ページではScholar向けのメタタグ（citation_title, citation_author, citation_publication_date等）を付与し、Google Scholarへのインデックスを促進する。
- sitemap.xmlとrobots.txtを自動生成する。
- hreflang属性による日英ページの言語関連付けを行う。
- canonical URLを全ページに設定し、重複インデックスを防止する。

### 4.4 パフォーマンス対策

- 画像のWebP変換・レスポンシブ画像（srcset）の自動生成。
- CSSのPurgeCSSによる未使用スタイル削除、HTML/CSS/JSのminify処理。
- フォントのfont-display: swap指定によるFOIT防止。フォントファイルのサブセット化。
- 遅延読み込み（loading="lazy"）の全画像適用。
- HTTP/2 + Brotli圧縮（Cloudflare Pages使用時に自動適用）。
- 目標値：Lighthouse Performance Score 95以上、First Contentful Paint 1.0秒以下。

## 5. データ・コンテンツ管理

### 5.1 コンテンツのデータ構造

各コンテンツカテゴリのデータ項目を以下に定義する。

**ニュース記事（content/news/YYYY-MM-DD-slug.md）：**
```yaml
---
title: "論文タイトルまたはニュース見出し"
date: 2026-04-01
categories: ["論文発表", "受賞", "イベント", "メンバー"]
tags: ["graphene", "quantum-dot"]
draft: false
---
本文をMarkdownで記述
```

**メンバーデータ（data/members.yaml）：**
```yaml
staff:
  - name_ja: "青木 伸之"
    name_en: "Nobuyuki Aoki"
    title_ja: "教授"
    title_en: "Professor"
    email: "aoki_at_faculty.chiba-u.jp"
    research_ja: "量子ナノデバイス、グラフェン量子輸送"
    research_en: "Quantum nanodevices, graphene quantum transport"
    photo: "/images/members/aoki.jpg"

students:
  - name_ja: "学生名"
    name_en: "Student Name"
    grade: "D3"
    year: 2024
```

**論文データ（data/publications.yaml）：**
```yaml
- title: "Signature of spin-resolved quantum point contact..."
  authors: ["Author A", "Author B", "N. Aoki"]
  journal: "Nano Letters"
  volume: "21"
  pages: "7534-7541"
  year: 2021
  doi: "10.1021/acs.nanolett.xxxxx"
  category: "journal"
```

**研究設備（data/facilities.yaml）：**
```yaml
- name_ja: "電子線描画装置"
  name_en: "Electron Beam Lithography System"
  category: "fabrication"
  description_ja: "ナノスケールパターニング用..."
  description_en: "For nanoscale patterning..."
  photo: "/images/facilities/ebl.jpg"
```

### 5.2 更新フロー

日常的なコンテンツ更新は、以下の2パターンのワークフローを用意する。

**パターンA：GitHubのWeb UIによる直接編集（推奨・非エンジニア向け）**  
1. GitHubリポジトリにブラウザでアクセスする。  
2. `content/news/` フォルダ内で「Add file > Create new file」を選択する。  
3. 所定のテンプレートに従いMarkdownファイルを作成する。  
4. 「Commit changes」ボタンを押すと、GitHub Actionsが自動でビルド・デプロイを実行する。  
5. 約1〜2分後にサイトに反映される。

**パターンB：ローカル編集 + Git push（エンジニア向け）**  
1. リポジトリをローカルにcloneする。  
2. テキストエディタ（VS Code推奨）でMarkdown/YAMLファイルを編集する。  
3. `hugo server`コマンドでローカルプレビューを確認する。  
4. `git commit` → `git push`でリモートに反映する。  
5. GitHub Actionsが自動ビルド・デプロイを実行する。

### 5.3 管理者権限

- **Owner（教員）：** リポジトリの全権限。設定変更、メンバー管理、デプロイ設定の変更が可能。
- **Maintainer（助教・博士課程上級生）：** コンテンツの直接編集・マージ権限。テンプレートやデザインの変更が可能。
- **Contributor（修士・学部生）：** プルリクエストの作成のみ。Maintainer以上の承認後にマージ・公開される。

### 5.4 コンテンツテンプレート

新規ニュース作成時のテンプレートファイルを `archetypes/news.md` として用意する。`hugo new news/YYYY-MM-DD-slug.md` コマンドでテンプレートが自動展開される仕組みとする。メンバーや論文の追加についてもREADMEにYAMLの記述例を明記し、コピー＆ペーストで追加できるようにする。

## 6. 移行・保守計画

### 6.1 既存コンテンツ移行方法

移行はフェーズに分けて段階的に実施する。

**Phase 1：コンテンツ抽出（1週間）**  
現行HTMLファイルからテキストコンテンツと画像ファイルを抽出する。HTMLをMarkdownに変換するスクリプト（Pandocベース）を用意し、手動による文体調整を最小化する。画像は原寸を保持しつつ、Hugoの画像パイプラインでWebP変換を行う。研究内容ページの図版（SEM像等）はキャプション情報と共にfront matterで管理する。

**Phase 2：テンプレート構築・データ整備（2週間）**  
Hugoテーマの作成（共通レイアウト、ナビゲーション、フッター）。メンバー情報・論文リスト・設備データをYAML化する。多言語ファイルの整備（i18nディレクトリの翻訳キー定義）。

**Phase 3：実装・デザイン適用（2週間）**  
Tailwind CSSによるスタイリング。レスポンシブ対応。アクセシビリティ検証。SEOメタデータの設定。Pagefind検索機能の統合。

**Phase 4：テスト・レビュー（1週間）**  
コンテンツの正確性確認（教員によるレビュー）。クロスブラウザ・クロスデバイステスト。Lighthouseによるパフォーマンス計測。アクセシビリティ監査（axe DevToolsによる自動チェック）。

**Phase 5：公開・切替（1〜2日）**  
DNSまたはサーバーパス設定の切替。旧URLからのリダイレクト設定。公開後1週間の監視期間を設け、アクセスログの確認とリンク切れチェックを実施する。

### 6.2 テスト戦略

- **コンテンツ正確性テスト：** 現行サイトの全ページと新サイトのテキスト内容を突合し、情報の欠落や誤りがないことを確認する。
- **ビジュアルリグレッションテスト：** Percy等のツールを用いて、デスクトップ/タブレット/モバイルの3ビューポートでスクリーンショット比較を行う。
- **リンクチェック：** 全内部リンク・外部リンクの生存確認をCI内で自動実行する（htmltest等）。
- **パフォーマンステスト：** Lighthouse CI をGitHub Actionsに組み込み、ビルド毎にスコアが閾値を下回った場合にアラートを出す。
- **アクセシビリティテスト：** axe-core による自動テストをCIに組み込む。WCAG 2.1 AA違反が検出された場合はビルドを失敗させる。

### 6.3 運用体制

- **日常更新：** 研究室の学生（修士以上）が担当。ニュース投稿、メンバー情報の更新、論文追加を行う。更新手順書（操作マニュアル）をリポジトリのREADMEおよび `/docs/` フォルダに整備する。
- **年次メンテナンス：** 年度初め（4月）にメンバー情報の一括更新、卒業生の進路先追加を実施する。年度末にニュースアーカイブの整理を行う。
- **技術メンテナンス：** Hugo本体やTailwind CSSのメジャーバージョンアップ対応は年1〜2回を想定。Dependabot（GitHub）による依存関係の自動更新通知を有効化し、セキュリティパッチの適用漏れを防止する。
- **引き継ぎ：** リポジトリのREADMEに環境構築手順、コンテンツ更新手順、デプロイの仕組みを全て文書化する。メンバー交代時の引き継ぎコストを最小化するため、技術選定は広く普及した標準的なツールに限定している。

### 6.4 バックアップ・障害対応

- Gitリポジトリ自体がコンテンツの完全なバックアップとして機能する。GitHub上のリポジトリが万一アクセス不能になった場合でも、ローカルクローンから復旧可能。
- 静的サイトのため、サーバーダウンやDB障害のリスクがない。CDNレベルの障害時はCloudflareのステータスページで確認し、復旧を待機する。
- 旧サイトのHTMLファイルはアーカイブとして `/legacy/` ブランチに保存し、必要に応じて参照可能とする。

## Skill routing

When the user's request matches an available skill, ALWAYS invoke it using the Skill
tool as your FIRST action. Do NOT answer directly, do NOT use other tools first.
The skill has specialized workflows that produce better results than ad-hoc answers.

Key routing rules:
- Product ideas, "is this worth building", brainstorming → invoke office-hours
- Bugs, errors, "why is this broken", 500 errors → invoke investigate
- Ship, deploy, push, create PR → invoke ship
- QA, test the site, find bugs → invoke qa
- Code review, check my diff → invoke review
- Update docs after shipping → invoke document-release
- Weekly retro → invoke retro
- Design system, brand → invoke design-consultation
- Visual audit, design polish → invoke design-review
- Architecture review → invoke plan-eng-review
- Save progress, checkpoint, resume → invoke checkpoint
- Code quality, health check → invoke health
